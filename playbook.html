What We're Building

When someone exits, changes roles, or transitions projects, critical institutional knowledge walks out the door — active projects, stakeholder relationships, undocumented systems, service accounts, tribal knowledge. Today this is handled inconsistently (if at all) through ad-hoc docs and rushed meetings.

The elevator pitch

An AI agent that conducts a structured conversational interview to extract what ISN'T documented, synthesizes it into a comprehensive handoff document, and publishes it to Confluence — automatically. Built with the Claude SDK with persistent memory so it gets smarter over time. Applicable across all of G&A, not just GTS.

How the Agent Works

Five steps from "enter your email" to a published wiki page. Each step maps to something one of us will build on Wednesday.

Step 1: Someone enters their email

We look up everything they've written — docs, wiki pages, their role and team — using enterprise search (Glean). Like a doctor pulling your chart before the appointment.

Step 2: The AI gets prepped

All that context plus a structured interview question tree gets fed to Claude. Now the AI knows who you are, what you've worked on, and what to ask.

Step 3: A guided conversation

10–15 turn chat. The AI asks smart questions, references their actual docs, follows up when something interesting comes up. A progress bar tracks coverage.

Step 4: A handoff doc is generated

A separate AI call takes the full transcript and produces a structured handoff document — responsibilities, risks, undocumented processes, contacts, open questions.

Step 5: Published to Confluence

Preview the doc, click "Publish," Confluence page appears. Done. 15 minutes of their time.

The bigger play: People don't have to be leaving to use this. Quarterly brain dumps build a living knowledge profile over time. When someone does leave, the handoff is already 80% done before anyone gives notice. This isn't just an exit tool — it's an ongoing knowledge capture system for the entire org.

Working With AI — Tips That'll Save You Hours

Whether you've been using AI tools for a while or this is your first time, these are the patterns that separate fast, clean results from spinning your wheels.

1. Always start in planning mode

Never jump straight into building. Both Claude Code and Codex have a planning mode — hit Shift+Tab before typing your prompt to activate it. In planning mode, the AI thinks through the approach and asks you questions before writing any code. This lets you catch bad assumptions before it writes 200 lines in the wrong direction.

✗ Bad: "Build me a chat interface"

✓ Good: Hit Shift+Tab first, then: "I need to build a chat interface. What questions do you have about what I want?"

2. Feed it context, not just commands

AI tools work dramatically better when you give them reference material.

→ Building something visual? Search for an example of something similar first. Otherwise the AI will pick a random design direction.

→ Working with an API or technical system? Find the official documentation and paste it in. The AI can read docs faster than you can — your job is to find the right docs and hand them over.

→ Working on content? Show it examples of good output. "Make it look like this" beats "make it look good" every time.

The golden rule: the AI doesn't know what's in your head. The more context you give it, the less it has to guess. And every guess is a chance for it to go sideways.

3. If it's stuck, switch — don't loop

Copy and paste the error. Don't describe it from memory. The exact error message matters.

The 3-try rule: If the AI can't solve a problem in 3 attempts, it's stuck in a loop. Switch to a different model or platform. A fresh model with fresh context often cracks it immediately.

Pro move: Keep a debug.md file listing what you've already tried. When you switch to a new model, hand it the debug log and say "don't retry anything in here."

4. Let planning mode break things down for you

You don't need to know how to decompose a big task into small steps — that's literally what planning mode does. Hit Shift+Tab, describe what you want at a high level, and the AI will break it into a sequence of steps and ask you questions.

Your job is just to review the plan. Does it make sense? Is it missing something? Tell it to adjust. Once the plan looks right, let it execute step by step.

Pro tip: If the AI gives you a plan, save it or ask it to write it to a file. That way you have a record of what was supposed to happen.

5. You're the pilot, not the passenger

The AI is going to give you stuff. Sometimes it'll be great, sometimes it'll be way off. That's normal. Just read through what it gives you, and if something doesn't look right, tell it. "Hey, that's not what I was going for" or "Try a different approach" is totally fine — it won't be offended.

And if you've been going back and forth for a while and it's not clicking, don't be afraid to just start a fresh conversation. Sometimes a clean slate with better context gets you there in 5 minutes.

6. When in doubt, show — don't tell

Screenshots, error logs, example code, sample outputs, URLs to documentation — anything concrete you can paste or show the AI will outperform a verbal description.

7. If the answer feels wrong, tell it to search the web

AI models run on training data — and training data gets stale. If the AI confidently tells you something that doesn't match what you're seeing, or gives you an answer that just feels off, tell it: "Search the web to verify that." Most AI coding tools (Claude Code, Codex, Cursor) can search the internet when you ask them to.

This is especially important for anything that changes frequently — API docs, library versions, framework syntax, pricing pages, or recent announcements. The AI might be working from information that was accurate 6 months ago but isn't anymore.

✗ Bad: Trusting the AI when it says "use api.v2.create()" without checking — that method might have been renamed or removed.

✓ Good: "That doesn't look right. Search the web for the current API docs and verify."

Rule of thumb: if the AI's answer involves a specific version number, API endpoint, or syntax you haven't seen before — ask it to verify with a web search before you build on it.

8. Git is your save button — let Claude handle it

You don't need to know git. You just need to know that it's your undo button. Here's the workflow:

When you start working, tell Claude: "Create a new branch called feature/[your-feature-name] and work in a worktree." This gives you your own isolated workspace where nothing you do can break anyone else's work.

As you build, tell Claude: "Commit this." Every commit is a save point. If something breaks, you can tell Claude "undo that last change" or "go back to the last working version" and it'll handle it.

When you're done, tell Claude: "Create a pull request for this." That packages up your work and sends it to the Captain to wire into the main project. You don't need to know what a pull request is — just that it's how your finished work gets delivered.

Think of it this way: Git = save states in a video game. Commits = quicksave. Branches = different save slots. Pull requests = handing your save file to the team lead. Claude handles all the commands — you just tell it when to save and when you're done.

TL;DR — The cheat sheet

1. Plan before you build. Shift+Tab for planning mode.
2. Feed it docs, examples, and screenshots — not just descriptions.
3. Stuck after 3 tries? Switch models. Keep a debug log.
4. Use planning mode (Shift+Tab) to break down big tasks for you.
5. You're the pilot. If something looks off, say so. Fresh conversations are free.
6. Show, don't tell. Paste the actual error. Screenshot the actual screen.
7. Answer feels wrong? Tell the AI to search the web — its training data goes stale.
8. Git = your save button. Tell Claude to commit often and create a PR when done.
